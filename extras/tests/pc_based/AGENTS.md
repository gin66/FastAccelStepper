# PC-Based Tests Agent Guidelines

## RMT Fill Buffer Testing (test_18)

### Context
The `rmt_fill_buffer()` function converts queue commands (steps + ticks) into RMT hardware symbols for ESP32. Each RMT symbol is a 32-bit entry containing two 16-bit sub-entries (low first, high second). Each sub-entry has:
- Bit 15: Step signal level (1=HIGH, 0=LOW)
- Bits 14-0: Duration in ticks

### Key Constraints
- `queue_entry.ticks` is `uint16_t`, max value 65535 (4095us at 16MHz)
- For periods >65535 ticks, ramp generator uses pause commands
- `ticks=65535` is special case (uses 2 RMT entries per step)
- ESP32C6 has PART_SIZE=24, other variants use 23 or 31
- Minimum RMT symbol period is 2 ticks (hardware limit)

### Test Validation Criteria
1. **Step count**: Number of rising edges in RMT output must match command steps
2. **Total ticks**: Sum of all RMT symbol durations must match command ticks
3. **High/Low ratio**: For step commands, high and low phases should be ~1:1 (50% duty cycle)
4. **Min symbol period**: Must be >= 2 ticks for hardware compatibility

### Reported Hardware Issues
- v=1000us (16000 ticks) => no pulses on real hardware
- v=2000us (32000 ticks) => no pulses on real hardware  
- v=500us (8000 ticks) => pulses but incorrect pattern (2:1 high:low ratio)

### Using ramp_helper for Test Cases
Generate realistic command sequences with:
```bash
make ramp_helper
./ramp_helper <speed_us> <acceleration> <steps>
```
Example: `./ramp_helper 1000 100000 5`

The output shows actual commands generated by the ramp generator, including step+pause pairs for longer periods.

### Test Structure
- Part 1: Fixed command inputs (single steps, multi-steps, pause combinations)
- Part 2: Step count limits (0-255 steps at various tick values)
- Part 3: High/Low ratio validation

### Common Issues
- Tick values >65535 overflow uint16_t (wrap around)
- Stretching code may fail with very small tick values (<96 for PART_SIZE=24)
- Special case for ticks=65535 uses different RMT encoding
